## Current problem

### Symptoms
- Images upload successfully to the backend.
- The metadata fetch from Supabase (`portfolio_images`) returns data with `imageUrl`.
- The browser displays the image as broken.
- Direct access to the saved image URL returns `404 Not Found`.

### Root cause
- The application is saving a public URL that points to a file that does not exist in the configured R2 bucket.
- Example broken URL:
  - `https://pub-bb51c92ec25541b7bac2969eeb40169b.r2.dev/uploads/admin_admin/Bedroom/1774994073035-6bc1243b-4d7d-440c-b601-523b2309afb3.png`
- Wrangler confirms the object key is not present in `theurbann-gallery` or `urbann-images`.

### What this means
- The upload process may be writing to a different bucket or a different key than the URL indicates.
- The public URL is correct for the R2 domain, but the object key is missing.
- This is not a Supabase metadata issue; it is an R2 object storage issue.

## Checklist to resolve
1. Verify the worker is using the same bucket configured in `wrangler.toml`.
2. Verify `R2_PUBLIC_URL` is set in the active environment and matches the public domain.
3. Confirm the file object exists in the bucket with the exact path saved to the URL.
4. Restart the Cloudflare Pages dev server after changing config.
5. Re-upload an image and inspect the `/api/upload` response back in the browser.
6. If object path is still missing, compare the filename/key generated by the backend with the public URL.

## Notes
- The current broken-image issue is not caused by the frontend fetch of Supabase data.
- It is caused by either:
  - Wrong object path in R2,
  - Wrong bucket binding,
  - or an upload target mismatch between local and remote R2.

## Recommended next step
- Re-upload one image and capture the upload response JSON.
- Use the exact `imageUrl` from that response to verify via browser or `curl`.
- If the object still 404s, the upload path/key is incorrect in R2.
